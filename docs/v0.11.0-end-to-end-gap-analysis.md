# Deep Agent v0.11.0 端到端差异审计

## 1. 审计范围与事实边界

本文比较两份固定事实：

- 参考项目快照：`/tmp/claude-code-reference-full-b17913e`
- 参考提交：`b17913e26fd4278ad5cd4b32ed3bde86bf1444e9`
- 当前项目：本文所在的 Deep Agent v0.11.0 工作树

参考项目只用于确认已经存在且能够工作的机制。本文不会把 Deep Agent 自己设计的行为冒称为参考项目原生实现，也不会因为两个项目目标相近，就声称两者逐行或逐参数等价。

本文使用以下标签：

- **参考原生**：固定参考快照中可以直接定位到的机制。
- **本地吸收**：Deep Agent 吸收了参考机制的原则，但使用 Python、DeepSeek 协议和本地安全边界重新实现。
- **本地设计**：固定参考快照中不存在对应实现，属于 Deep Agent 自己的产品或可靠性决策。
- **明确差异**：两边当前仍有实质性行为差异，不能在发布说明中写成“已完全对齐”。

## 2. 结论摘要

Deep Agent 当前已经可靠吸收了以下参考原则：

1. Agent Loop 根据实际结构化工具调用继续，而不是只相信模型的 finish reason。
2. 大型工具结果先持久化或缩减，再进入模型上下文。
3. 工具并发必须按解析后的输入显式判定安全；Deep Agent 进一步采用“仅纯读并发、写入和状态变更串行”的更严格本地策略。
4. 中断后必须为每个 tool call 保留或合成配对的 ToolResult。
5. 自动压缩失败要有熔断，不能无界重复调用模型。
6. 终端截断必须按 grapheme cluster 和显示宽度处理。
7. Read 输出的行号前缀必须与源码缩进有不可歧义的边界。

以下行为是 Deep Agent 的本地设计，不是参考项目原生机制：

- 4/8/16/24 工具轮次软目标与默认 32 轮硬上限；
- 为实现和验证预留轮次的 soft/hard phase；
- tool-free 独立 final synthesis；
- DeepSeek DSML 文本防御和异常 finish reason 纠正；
- Runtime 计划硬完成门；
- managed `run_tests` 自动选择器；
- 确定性 emergency context projection。

当前仍未等价的主要部分是：

- Resume 没有恢复参考项目式的可重建消息链，也没有重放 tool-result replacement decision；
- JavaScript/TypeScript package script 仍固定通过 `npm run` 启动；
- 网络重试仍缺少 Retry-After、jitter、query-source 策略、长等待 heartbeat 和显式逐 chunk idle watchdog；
- 通用 `verify` 步骤是否完成仍主要依赖模型调用 `agent_update_step`，只有特定 artifact 流程有更强的工具证据门。

## 3. Agent Loop

### 3.1 参考原生机制

参考入口是 `/tmp/claude-code-reference-full-b17913e/src/query.ts` 中的 `queryLoop`：

- `while (true)` 驱动连续轮次；
- 流中收集到的 `toolUseBlocks` 是是否继续执行工具的实际信号；
- 源码明确说明 `stop_reason === 'tool_use'` 不可靠；
- 没有工具块时，经过 stop hooks、token budget 等收口逻辑后结束；
- 可选 `maxTurns` 在工具结果批次完成后增加并检查 `turnCount`；中断工具批次也会检查下一计数。

`maxTurns` 不是所有 `while` 迭代或 API 请求的总上限：Stop hook blocking 和 token-budget continuation 会保留同一个 `turnCount`。参考位置为 `src/query.ts:1282-1340,1506-1515,1678-1711`。

这说明参考项目的核心不是某个固定轮次数值，而是“实际工具块存在则继续，没有工具块才进入完成路径”。

### 3.2 Deep Agent 当前实现

当前入口是 [`agent/runtime.py`](../agent/runtime.py) 中的 `AgentRuntime._execute`：

- 以响应中的结构化 `tool_calls` 决定是否执行工具；
- 工具轮次与模型请求轮次分开计数；
- 工具轮次达到 soft target 后，只有统一执行证据门已满足才进入收口；
- Task Graph、真实非计划工具、显式单次验证或指定 artifact 证据缺失时，会聚合提示并继续到 hard limit；
- 异常 finish reason 的响应不会执行其中的工具调用，只允许一次有界纠正；
- 进入 hard phase 后会收缩可用工具 schema，保留实现、验证和计划迁移所需能力。

### 3.3 相同、差异与风险

相同点是两边都依据实际工具调用结构，而不是单独依据 finish reason。不同点是 soft target、hard limit、阶段 schema 过滤、计划迁移门和工具轮次计数都是本地设计。

因此文档可以写“参考了以实际工具调用驱动循环的原则”，不能写“轮次策略照搬参考项目”。当前主要风险是复杂计划仍可能一直运行到 hard limit；这时程序应如实失败并保存 Resume，而不能把达到上限包装成完成。

## 4. 轮次终止、完成判定与 final synthesis

### 4.1 参考原生机制

参考 `queryLoop` 在没有 `toolUseBlocks` 时进入 stop hook、token budget 和完成路径；`maxTurns` 只约束完成工具结果批次后推进的 turn，不约束同一 turn 内的全部无工具续轮。参考主循环没有 Deep Agent 当前这种独立 final synthesis，也没有 DeepSeek DSML 协议文本。

### 4.2 Deep Agent 当前实现

[`agent/runtime.py`](../agent/runtime.py) 中的关键本地机制包括：

- `_completion_issue`：拒绝空答案、短进度说明和未完成的必需 Task Graph；
- `_final_synthesis`：工具预算关闭后进行一次 tool-free 用户答复合成；
- `_tool_protocol_violation`：同时用于主循环与 final synthesis，拒绝结构化工具调用和已知 DSML 工具协议文本；
- `_has_usable_finish_reason`：异常 finish reason 不会直接进入工具执行或成功终态；
- `_incomplete_answer`：缺少实现或验证时保存失败 Session，并给出 Resume 命令。

已知 DSML 防御覆盖两类输入：

1. 响应仍包含结构化 `tool_calls`；
2. 响应正文包含 DeepSeek 的 `DSML tool_calls` 或 `DSML invoke` 标记。

这些响应不会被当成用户答案，也不会在 tool-free final synthesis 中偷偷执行工具。

### 4.3 定性

final synthesis、soft/hard 轮次和 DSML 防御全部是 **本地设计**。它们解决的是 DeepSeek 兼容协议和 Deep Agent 完成语义，不能写成参考项目原生功能。

剩余风险是 DSML 文本检测只能覆盖结构化调用和当前已知标记；未知的纯文本协议变体仍需由空答、计划门和用户可见失败语义兜底。

## 5. 工具结果预算、附件与读取闭环

### 5.1 参考原生机制

参考文件包括：

- `/tmp/claude-code-reference-full-b17913e/src/constants/toolLimits.ts`
- `/tmp/claude-code-reference-full-b17913e/src/utils/toolResultStorage.ts`

固定参考中的主要上限与机制是：

- 默认单工具结果阈值为 50,000 字符；
- 启用 `tengu_hawthorn_steeple` gate 时，单个 API user message 内的 tool result 聚合目标为 200,000 字符；
- `persistToolResult` 把大型文本结果保存到 Session 目录；
- `buildLargeToolResultMessage` 给模型返回路径和预览；
- `enforceToolResultBudget` 按 message 分组，并冻结已经做出的替换决定；
- `recordContentReplacement` 把新 replacement 写入 transcript；
- `reconstructContentReplacementState` 在 Resume 时逐字恢复 replacement，保持 prompt cache 和历史语义稳定。

这里的“聚合预算”不是无条件硬上限：该 gate 默认关闭；已经被模型看过而冻结的结果本身超限时，参考实现接受暂时超限并等待后续 microcompact。大型结果持久化失败时，`maybePersistLargeToolResult` 会原样返回结果，属于 fail-open。参考的目录是 project/session 下的本地 `tool-results`，源码没有 Deep Agent 当前的权限 mode、符号链接、数量、总字节和内容哈希元数据，因此不能把两者都概括为同等级的“私有安全附件”。对应位置为 `src/utils/toolResultStorage.ts:94-183,267-333,447-455,641-691,924-987`。

### 5.2 Deep Agent 当前实现

当前实现位于：

- [`agent/tools/result_store.py`](../agent/tools/result_store.py) 的 `ToolResultStore`
- [`agent/tools/manager.py`](../agent/tools/manager.py) 的 ToolResult 持久化入口和 `tool_result_read`
- [`agent/convergence.py`](../agent/convergence.py) 的 `ToolHistoryCompactor`

当前默认值包括：

- 12,000 bytes 以上落附件；
- 8 MiB 单附件上限；
- 512 个附件和 256 MiB 单 Session 上限；
- 单次 `tool_result_read` 最多读取 32,000 字符；
- 附件目录和文件使用私有权限，并校验普通文件、符号链接、request ID、数量和总字节；读取时重新计算并返回内容 SHA-256。

这些是比参考项目更严格的本地安全和资源边界，不应删除来追求表面一致。

### 5.3 hard phase 的验证附件例外

大型 TypeScript 验收曾暴露以下闭环缺陷：`run_tests` 生成大型附件后，hard phase 同时关闭了 `tool_result_read`，模型只能看到 preview，无法继续读取刚产生的验证证据。

当前工作树已经加入一个 **本地受限修复**：

- 只在 `implement` 或 `verify` 进行中开放；
- 只接受当前 Session 中由 `run_tests`、LSP diagnostics、document parse、staged diff verification，或非文件探索型 shell validation 产生的附件；
- request ID 必须与已记录 ToolResult 的附件 ID 一致；
- 每个读取块最多 12,000 字符；
- 每个 Session turn 默认最多两次，配置硬限为四次；
- 普通探索附件仍在 hard phase 关闭。

这个例外解决了“验证刚完成却无法查看验证结果”的闭环，同时没有重新开放一般文件探索。它不是参考项目原生机制，而是 Deep Agent soft/hard phase 下必须配套的本地设计。

### 5.4 剩余差异

当前附件可以按 request ID 恢复已持久化的完整或明确标记的有界 ToolResult，但还没有参考项目那种“replacement decision 写 transcript、Resume 时 byte-identical 重放”的机制。

另外，`ToolHistoryCompactor` 的注释不能把 AgentState 描述为完整正文仓库：ToolManager 会先把大结果换成 preview，然后 Runtime 才记录 AgentState。附件只在结果未超过 8 MiB 硬限、上游未先截断且持久化成功时保存对应正文；超硬限、上游截断或持久化失败时，只保留有界正文、哈希和明确的不可用标记。

## 6. 微压缩、自动压缩、熔断和溢出恢复

### 6.1 参考原生顺序

参考 `/tmp/claude-code-reference-full-b17913e/src/query.ts` 暴露的 feature-gated 调用顺序是：

1. tool-result budget；
2. history snip；
3. microcompact；
4. context collapse；
5. autocompact。

其中 `snipCompact.ts`、`reactiveCompact.ts`、`contextCollapse/*` 和 `cachedMicrocompact.ts` 并不在该固定公开副本中，因此上述顺序只能证明接入点，不能宣称所有 gate 在该副本内都有可运行实现。可直接核对的 `autoCompact.ts` 还提供：

- 在两个默认关闭的 gate 同时启用时，先尝试 `trySessionMemoryCompaction`，失败或不适用才进入传统摘要；Deep Agent 没有对应的 SessionMemory shortcut；
- 最多 20,000 output tokens 的 summary 预留；
- 连续三次自动压缩失败后熔断；
- 成功后清零失败计数；
- `query.ts:1065-1175` 的 413/reactive single-shot guard，避免同一 query loop 重复恢复；reactive 实现模块本身不在固定公开副本中。

### 6.2 Deep Agent 当前实现

当前主要实现位于 [`agent/convergence.py`](../agent/convergence.py) 和 [`agent/runtime.py`](../agent/runtime.py)：

- `ToolHistoryCompactor` 对旧 ToolResult 做 head/tail preview、metadata 压缩和确定性 hard-limit fallback；
- `ContextWindowController` 估算请求、输出预留和安全缓冲；
- `_auto_compact_context` 用 DeepSeek 生成 continuation summary；
- 连续三次失败后打开 semantic-compaction circuit；
- circuit 跨 Resume 保留，且 provider overflow 的 semantic stage 也会尊重 circuit，直接进入确定性 fallback；
- `_emergency_context_collapse` 在模型压缩不可用时生成受限的确定性状态投影；
- tool call 与 ToolResult 配对在压缩前后都会修复。

### 6.3 相同、差异与风险

“先缩减工具历史、再做语义压缩”和“连续失败后停止重试自动压缩”有清楚的参考依据。参考 `autoCompact.ts:257-265,334-349` 在失败时只累计计数并熔断，没有 Deep Agent 这种自动压缩失败后的确定性 emergency projection；该 fallback 是本地设计。以下数值和实现也属于本地设计：

- 96K 工具历史 envelope；扣除 24K 输出预留后，默认实际压缩目标为 72K；
- 48K 同轮结果预算；
- 12K 单结果模型可见预算；
- 1.2K 压缩预览；
- 65,536 token 默认窗口与 8,192 token safety buffer；
- DeepSeek 自摘要和 emergency projection 格式。

普通一次语义压缩失败不会无条件立刻投影：只有请求仍超过 hard input limit，或 circuit 打开后仍超过 proactive trigger，才进入确定性收缩。emergency projection 只保留受限状态、哈希和少量 excerpt，不能被描述为完整工具正文。semantic-compaction circuit 跨 Resume 保留；熔断打开时，主动路径和 provider-overflow semantic stage 都跳过新的模型压缩请求，改用确定性 fallback。

## 7. 工具编排、并发和中断配对

### 7.1 参考原生机制

参考文件包括：

- `/tmp/claude-code-reference-full-b17913e/src/services/tools/toolOrchestration.ts`
- `/tmp/claude-code-reference-full-b17913e/src/services/tools/StreamingToolExecutor.ts`
- `/tmp/claude-code-reference-full-b17913e/src/query.ts` 的 `yieldMissingToolResultBlocks`

参考行为是：

- 只有工具的 `isConcurrencySafe(input)` 明确返回 true 才并发；
- 输入解析失败或安全判定抛错时保守串行；
- 非并发工具独占执行；该判定不等同于“纯读”，参考中的 TaskCreate、TaskUpdate、Agent 和 Config 等有状态工具也可以按自身输入返回并发安全；
- streaming executor 维护 queued、executing、completed、yielded 状态；
- 非流式并发路径通过 `Promise.race` 按完成先后产出；流式 executor 也可以越过尚未完成的并发安全前项；
- sibling error 和用户中断可以生成 synthetic ToolResult；streaming fallback 则 tombstone 旧 assistant 消息、丢弃旧 executor 并清空旧结果；
- query 异常或中断时补齐缺失的 tool result block。

### 7.2 Deep Agent 当前实现

当前实现位于 [`agent/tool_orchestration.py`](../agent/tool_orchestration.py)：

- Capability 必须显式 `concurrency_safe=True`；
- permissions 必须精确为纯 `read`；
- 需要确认、不可用、被 Runtime 拒绝或有状态的 `file.diff` 都是串行屏障；
- 默认最多四个并发安全读取，硬上限十六；
- 正常和异常结果都保持原 tool call 顺序；
- Ctrl+C 时保留已经完成的结果，为未完成和未开始调用生成 synthetic result；
- Runtime 先 checkpoint 完整配对批次，再重新抛出原中断；
- 命令执行器会终止完整子进程组后再传播中断。

### 7.3 定性与风险

这是当前与参考最接近的一组机制，但仍不是相同实现：参考是按输入动态判定并支持 streaming tool execution，且并发结果的可观察产出顺序不保证等于模型调用顺序；当前使用静态 Capability 元数据和 Python ThreadPool，并在完整批次返回时显式恢复模型调用顺序。参考证据位于 `src/services/tools/toolOrchestration.ts:86-176`、`src/utils/generators.ts:31-65`、`src/services/tools/StreamingToolExecutor.ts:407-455` 和 `src/query.ts:709-739`。

当前没有参考 executor 的逐工具 `interruptBehavior=cancel/block`。Ctrl+C 后少量纯读 Future 可能仍在后台退出，但它们被限制为显式并发安全的纯读工具，写工具不会越过串行屏障继续执行。

### 8.2 Deep Agent 当前实现

当前相关实现包括：

- [`agent/state.py`](../agent/state.py) 的 `plan_step_satisfied` 和 `can_skip_plan_step`；

当前规则是：

### 8.3 明确差异

因此可以写“计划未完成时禁止成功收口”，不能写“每个 completed 步骤都已由 Runtime 独立证明”。

## 9. 测试命令选择

### 9.1 参考事实

固定参考快照没有一个通用的 managed `run_tests` 自动选择器。它自己的 `package.json:10-14` 声明：

```json
{
  "scripts": {
    "build": "bun run scripts/build.ts",
    "dev": "bun run src/entrypoints/cli.tsx",
    "typecheck": "bun x tsc --noEmit"
  }
}
```

参考项目的 bundled instructions 会提示 Agent 阅读 README、manifest 和现有脚本，但这不是自动选择器实现。

### 9.2 Deep Agent 当前实现

[`agent/tools/templates.py`](../agent/tools/templates.py) 的 `run_tests`：

- 优先识别 `pyproject.toml`、`pytest.ini`、`package.json`、Cargo、Go、Gradle 和 Maven；
- 不再因为存在一个泛用 `tests/` 目录就把 JavaScript/TypeScript 项目误判成 pytest；
- 对 package 项目只允许已经存在的 `test`、`typecheck`、`check`、`lint`、`build`；
- 不会盲目执行不存在的 `npm test`；
- 命令通过分离 argv 进入受限进程执行器。

### 9.3 明确差异与风险

当前 `_run_package_script` 仍固定执行：

```text
npm run <script>
```

它没有读取 `packageManager` 字段，也没有根据 `bun.lock`、`pnpm-lock.yaml`、`yarn.lock` 选择 runner。本次固定参考快照中，npm 可以调用脚本内的 `bun x tsc`，所以这不是大型 TypeScript 验收失败的直接根因；但在没有 npm 的 Bun 项目或要求严格 runner 的项目中仍会失败。

managed `run_tests` 是 **本地设计**，不能写成参考项目原生功能。

## 10. 网络重试与流式恢复

### 10.1 参考原生机制

参考文件：

- `/tmp/claude-code-reference-full-b17913e/src/services/api/withRetry.ts`
- `/tmp/claude-code-reference-full-b17913e/src/services/api/claude.ts`

参考机制包括以下有条件的策略：

- 解析数字秒形式的 Retry-After；
- 指数退避加随机 jitter；
- backoff sleep 可由 AbortSignal 中断；
- foreground/background query source 使用不同的 529 策略；
- 529 计数、熔断或模型 fallback 会根据 query source、模型、订阅、环境和可用 fallback model 决定；
- unattended feature 与 persistent mode 启用时，长等待按 30 秒切块并发送 heartbeat；
- `CLAUDE_ENABLE_STREAM_WATCHDOG` 启用时，streaming idle watchdog 在长时间没有 chunk 时主动终止流；
- env/GB gate 启用时禁止 streaming-to-non-streaming fallback，以降低重复工具副作用风险；默认分支仍会发起 non-streaming fallback。

### 10.2 Deep Agent 当前实现

[`agent/deepseek.py`](../agent/deepseek.py) 当前提供：

- 同一 Key 上对 408、部分 5xx 和传输故障做有界指数退避；
- 401、403、429 切换下一个 DeepSeek Key；
- 网络尝试次数写入模型指标；
- 流式响应在第一个有效 delta 之前可以重试或退回普通请求；
- 一旦收到 reasoning、正文或工具调用 delta，就不再重放请求，而是抛出可 Resume 的 `DeepSeekStreamInterrupted`；
- HTTP 错误正文有大小限制和凭据脱敏；
- `urlopen` 有 socket/request timeout。

### 10.3 明确差异与风险

当前仍缺少：

- Retry-After；
- jitter；
- query-source retry policy；
- AbortSignal 风格的 backoff 取消；
- 长等待 heartbeat；
- 每次收到 chunk 都重置的显式 idle watchdog。

`urlopen(timeout=...)` 不是参考项目可选的主动 chunk watchdog。Deep Agent 当前采取“收到任一有效 delta 后不重放”的更保守本地策略；参考默认配置不提供同样的无条件保证。两边完整重试策略尚未等价。

## 11. Resume

### 11.1 参考原生机制

参考 Resume 不是只取最近几条元数据。它持久化并恢复一条可重建、带一致性遥测的消息链：

- `recordTranscript` 保存带 parent UUID 的 transcript；
- `buildConversationChain` 从 leaf 回溯，检测 cycle，并恢复并行工具产生的孤立 sibling/tool result；
- `applyPreservedSegmentRelinks` 重放压缩后的保留段连接；
- `applySnipRemovals` 按记录的 removed UUID 删除中间历史并跨 gap 重连；
- `checkResumeConsistency` 比较写入时 message count 与恢复链位置并发送遥测，但不拒绝恢复；
- `recordContentReplacement` 和 `reconstructContentReplacementState` 重放工具结果预算决定；
- `ResumeConversation.onSelect` 把恢复的 messages 和 content replacements 交给 REPL；context-collapse entries 在 `onSelect` 内直接调用 `restoreFromEntries` 恢复，不是 REPL prop。

这也不是无损保证：检测到 cycle 时实现记录日志并返回 partial chain；旧 snip 条目缺少 `removedUuids` 时可能恢复 pre-snip 历史。参考位置为 `src/utils/sessionStorage.ts:1976-1978,2069-2094,2208-2243` 和 `src/screens/ResumeConversation.tsx:264-297`。其中 context-collapse 的 `restoreFromEntries` 是 gated 接入点，相关实现模块不在固定公开副本中。

### 11.2 Deep Agent 当前实现

当前相关实现位于：

- [`agent/session.py`](../agent/session.py)
- [`agent/runtime.py`](../agent/runtime.py) 的 `_resume_locked`
- [`agent/context.py`](../agent/context.py) 的 `_resume_section`
- [`agent/prompt.py`](../agent/prompt.py) 的 `build_resume`
- [`agent/state.py`](../agent/state.py) 的 `resume`

Session 文件会保存消息列表和 AgentState，但 Resume 时不会把旧消息列表重新作为模型历史。`PromptBuilder.build_resume` 仍只生成三条新消息：系统 Prompt、ContextPackage 和当前用户要求。

Resume Context 当前投影：

- 原始目标；
- 当前 Resume 指令；
- 最近 12 条工具调用的类型、成功状态和安全目标元数据；
- previous assistant outcome；
- 最近错误。

### 11.3 已修复的本地死锁

失败验收时曾出现：模型丢失源码正文，却继承 `implementation_reads_used`、连续只读轮次和 hard phase，导致 Resume 后正规读取入口仍关闭。

当前 `AgentState.resume` 已把这些 gate 视为 per-turn 状态并重置：

- implementation evidence reads；
- validation attachment reads；
- consecutive read-only/low-yield rounds；
- nudge 和 hard notice 状态。

它仍保留 durable seen targets 和 semantic-compaction circuit。这样 Resume 可以重新获取有界证据，而不会把失败 turn 的探索配额永久带入下一 turn。

`state.start()` 会清空 `state.error`。当前 `_resume_section` 已改为回退读取 `execution_context.recent_error`，因此正常 v0.11.0 Session 不会再把 “Last Recorded Error” 错写成 `None`。

### 11.4 尚未等价的部分

上述修复打破了恢复死锁，但不等于参考项目的消息链恢复：

- 源码正文仍需重新读取；
- tool-result replacement decision 没有 transcript replay；
- compact/snip 边界没有 UUID 级恢复；
- 没有 write-to-load message-chain consistency check；
- 若文件在两次运行之间变化，当前 Resume 无法重现上一 turn 的精确证据视图。

因此发布说明只能写“Resume 可重新开放有界恢复窗口并保留任务状态”，不能写“Resume 已与参考项目无损等价”。

### 13.2 Deep Agent 当前实现

[`agent/tools/templates.py`](../agent/tools/templates.py) 的 `read_file` 现在始终输出：

```text
     1→source
     2→    indented source
```

旧实现使用“行号 + 两个普通空格 + 源码”，分隔空格与源码缩进无法区分。大型 TypeScript 验收中，模型因此把四空格源码复制成六空格，随后 `file_diff.old_text` 无法匹配。

当前箭头边界解决了这一根因。它与参考项目使用同一类“显式 tab-or-arrow 边界”，但不能写成逐字相同，因为参考默认优先 tab，当前固定使用 padded arrow。

[`agent/tools/file_edit.py`](../agent/tools/file_edit.py) 还会在搜索文件前拒绝 `old_text == new_text`，避免把无变化请求误报成 old text 未找到。

## 14. 大型 TypeScript 验收暴露的根因组

失败运行不是十个独立小问题，而是一条相互依赖的根因链：

1. `run_tests` 执行固定快照声明的 `typecheck` 脚本，生成约 600 KiB 的错误输出；
2. 固定研究快照本身缺少多个源码文件，因此数千条 typecheck 错误不能证明某一个候选文件存在独立真实 Bug；
3. 大型结果被正确持久化为附件，但当时 hard phase 禁止补读附件；
4. 精确源码补读使用歧义空格边界，模型复制出错误缩进；
5. 两次 `file_diff.old_text` 无法匹配；
6. Resume 又丢失正文并继承耗尽的读取 gate，无法确定性恢复。

在上述修复全部离线通过后，2026-07-15 又用全新 Workspace 进行了一次、也是本轮最后一次在线大型验收。该运行没有修改研究快照，最终中文报告已经满足源码证据、至少 8 项优点、Bug/限制、验证结果和剩余风险等语义检查，但 Session 仍被判为 `failed`。脱敏指标为：31 个有完整工具结果的工具轮、4 个没有形成完整工具批次的 main-loop 请求、1 次 final synthesis，共 36 次逻辑请求；没有 context-compaction 请求或 HTTP 重试。不能把这次失败写成在线通过。

第二次失败又暴露出同一层的两个确定性分类缺陷：

1. artifact 正则允许跨分号匹配，把“文件或逐条打补丁；不得读取或输出任何真实凭据”误判成要求输出文件；
2. “若没有充分证据，未找到可证实缺陷是合格结论，应跳过 implement、不要修改代码”没有被识别为 conditional mutation。

这导致第 25 轮合法的 `implement -> skipped` 被拒绝，随后计划管理继续消耗轮次；即使 final synthesis 已生成完整报告，虚假的 managed-write 门仍要求不存在的 `file_apply` 证据。

工具轨迹还证明模型没有遵守“只运行一次静态检查”：一次 `run_tests` 失败后又发起了一个 shell 验证和两次 LSP 诊断。旧 Runtime 只限制通用依赖/等价探针轮数，没有把用户明确的单次验证要求变成执行策略，因此内容报告虽然诚实列出了限制，仍实际重复了等价检查。

当前工作树对应状态：

| 根因 | 当前状态 | 定性 |
|---|---|---|
| Read 行号与缩进边界歧义 | 已改为显式箭头并有回归 | 直接吸收参考机制 |
| `old_text == new_text` 错误优先级 | 已在查找前拒绝并有回归 | 直接吸收参考机制 |
| hard phase 无法读验证附件 | 已加当前 Session、验证来源、次数和块大小限制 | 本地设计修复 |
| Resume 继承耗尽 gate | 已按 turn 重置 | 本地设计修复 |
| Resume Last Recorded Error 被清空 | 已回退到 ExecutionContext | 本地设计修复 |
| 主循环/final synthesis 接受 DSML 文本 | 已统一拒绝且不执行工具 | DeepSeek 本地设计 |
| artifact 正则跨分句误判否定输出 | 已限制在同一分句并逐候选过滤否定动作 | 本地分类根因修复 |
| 较长 conditional-mutation 表述漏判 | 已组合“无证据”和“skip/no-change”两个有界信号 | 本地分类根因修复 |
| 明确单次验证后仍重复 shell/LSP | 已增加 `single-validation` 路由、schema 收缩与 Runtime 拒绝 | 本地执行约束修复 |

| 目录/文件产物被无关写入、后续删除或撤销冒充完成 | TaskRoute schema 2 保存有界文件与目录 hints；按 active apply/delete/undo 和 Word preview lineage 判定 | 本地完成真实性修复 |

旧失败 Session 不能用来证明新代码已经通过。当前修复使用失败验收的原始关键句建立了确定性回归：路由必须得到 `conditional-mutation=true`、`artifact-required=false` 和 `single-validation=true`；无可证实缺陷时可跳过 implement，仍必须完成 verify；4 个工具轮后一次 tool-free final synthesis 可完成 Session，且没有 `file_apply`、managed-write 或 Resume 错误；第一次验证失败后，等价 shell 与 LSP 尝试在执行前被拒绝。对应证据位于 [`tests/test_v09_routing.py`](../tests/test_v09_routing.py) 和 [`tests/test_runtime_convergence.py`](../tests/test_runtime_convergence.py)。

由于本次在线运行已消耗约 147.7 万 token，且用户明确要求避免继续消耗余额，修复后不再立即重复在线验收。发布材料必须准确写成“在线失败已复盘，修复后确定性离线回归通过，未宣称修复后在线通过”。未来若经用户授权重跑，必须使用全新 Workspace/Session，并一次性记录扫描范围、请求/工具计数、修改证据、验证附件读取、计划终态和最终回答。

## 16. 当前优先级

1. 保留当前原始失败证据与确定性回归；只有在用户明确同意新的 API 成本后，才用全新大型 Workspace 做一次修复后在线验收，不能 Resume 旧失败 Session 冒充通过。
2. 为 Resume 设计有预算的可重建消息或 durable evidence segment，并重放工具结果 replacement decision。
3. 让 package runner 读取 `packageManager` 和 lockfile，同时保持 manifest 脚本 allow-list。
4. 为网络重试补 Retry-After、jitter、可取消 backoff、heartbeat 和显式 stream idle watchdog。
5. 为通用 verify 建立与任务类型匹配的工具证据规则，避免仅凭模型状态更新宣布完成。

这些后续项都应保持 `CLI -> Runtime -> AgentState -> Prompt -> Capability -> Permission` 和 `ToolRequest -> PermissionManager -> ToolResult` 边界，不得为了表面对齐绕过 Deep Agent 的安全架构。
