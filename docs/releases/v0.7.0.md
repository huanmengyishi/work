# Deep Agent v0.7.0

## Release reason

Version 0.6 established decision state. Version 0.7 makes code changes more
reliable and keeps long-running projects maintainable: immediate diagnostics,
bounded semantic context, Memory lifecycle control, compact resume, and an
optional background maintenance process.

## Included functionality

- LSPManager-style bounded diagnostics for Python via Pyright and JavaScript or
  TypeScript via `tsc --noEmit`.
- Independent engine degradation and automatic diagnostics after atomic
  `file_apply` without turning a successful write into a failed write.
- Semantic module summaries, exports, and internal import relationships in the
  optional `index.semantic.json` sidecar.
- Memory confidence, use counts, last-used time, expiry, merge lineage, duplicate
  maintenance, protected kinds, and `agent memory maintain [--apply]`.
- Compact resume context that discards unlimited raw tool transcripts while
  preserving the previous outcome, AgentState, Execution Context, project
  context, relevant Memory, and current capabilities.
- Optional per-project Daemon with polling, incremental context refresh, Memory
  maintenance, stale-PID handling, process identity checks, lock protection,
  atomic state, and bounded intervals.
- Queue cross-process locking, SQLite WAL/busy timeout, and FTS backfill for
  concurrent foreground and background operation.

## Claude Code reference decision

The `free/claude-code` Java/Spring AI rewrite was used as a design reference.
Deep Agent adopted bounded context, observable capability degradation, audited
resource boundaries, resumable state, and explicit tool loops. It did not adopt
a second Java runtime, multi-provider abstraction, full-screen TUI, JAR plugins,
arbitrary hooks, telemetry, or unrestricted skills/agents loading because those
choices conflict with the single-DeepSeek model and the existing
CLI -> Runtime -> AgentState -> Prompt -> Capability -> Permission boundary.

## Full code review findings and fixes

- Fixed Memory queries and statistics so merged records cannot reappear through
  SQL precedence, FTS limits, vector fallback, recent lists, or normal listing.
- Replaced transitive duplicate grouping with keeper-relative grouping to avoid
  merging records whose endpoints are below the configured similarity limit.
- Added automatic default expiry only for non-protected Memory kinds and parsed
  timestamps as timezone-aware values.
- Added SQLite WAL, 30-second busy timeout, and FTS rebuild for databases created
  before FTS triggers existed.
- Fixed LSP capability availability so one installed engine remains usable and
  excluded dependency/vendor directories from source discovery.
- Kept successful atomic writes successful when diagnostics report source errors.
- Enforced the total rendered Context limit after source and semantic summaries.
- Compacted Session resume to prevent unbounded Prompt growth.
- Added explicit Runtime/ToolManager close paths for REPL, queue, and one-shot
  operation without prematurely closing reusable interactive tools.
- Removed Daemon mutation of active queues, validated PID ownership before
  signaling, and moved Daemon/Memory output under the configured data directory.
- Added cross-process Queue locks and preserved task identity when a Parallel
  worker raises unexpectedly.
- Filtered Broken/Unavailable capabilities from both model schemas and Prompt
  summaries, and from `AgentState.loaded_tools`.
- Expanded event-log redaction for cookies, passwords, secrets, and API-key name
  variants.
- Made Workspace Memory tolerant of malformed dependency mappings and file I/O
  failures.

## Next direction

Version 1.0 remains a future optimization target only: freeze Runtime interfaces,
make ContextBuilder the sole context provider, complete the Event Bus side-effect
boundary, and keep all future integrations behind Capability Registry and
Permission Manager.
